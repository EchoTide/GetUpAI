# GetUpAI 桌面端：AI 功能重构计划（讨论版）

## 目标（共识）
- 把 AI 从“嘴炮式回复”升级为“监督代理（Coach Agent）”：有目标、有状态、有工具、有边界、可解释、可回滚。
- 把“我有理由”统一升级为“申诉/协商”：理由 → 决策 → 执行 → 复盘；不再只是聊天。

## 现状摘要
- 主界面“我有理由”：AI 仅生成一句回复，不改变系统策略与计划。
- 弹窗“我有理由”：AI 输出 accept/reject + pauseMinutes（已具备一点决策雏形，但浅、工具少、不可追踪）。
- 核心闭环已具备：监控 → 干预 → 记录 → 强制弹窗。

## 重构方向
### Phase 0：统一语义与入口
- 将“我有理由”统一命名与定位为“申诉/协商”，主界面与弹窗使用同一套决策输出（结构化）。
- 将“AI 老师/讲道理”定位为服务策略决策的交互层：人格可以保留，但输出必须能驱动可执行动作。

### Phase 1：将“申诉/协商”做成可执行的裁决单
- 输入：快捷理由（会议/写代码/电话/专注冲刺/吃饭/通勤/身体不适…）+ 可选补充文本。
- 输出（结构化裁决单）：
  - verdict：accept / reject / accept_with_conditions
  - pauseMinutes：建议暂停多久（带上限与今日预算）
  - tradeoff：补偿动作（暂停后强制站立/走动、提高后续提醒强度等）
  - explain：简短、可引用事实的解释
- 落地：裁决单必须能改变系统状态（pauseUntil/nextReminderAt/策略强度/当日预算），并可在日志中追踪。

### Phase 2：让 AI 真正成为策略引擎（工具 + 权限边界）
- 工具（先内部数据，安全）：
  - 读：今日/本周统计、最近 2 小时行为、忽略/申诉历史、休息时段、站立办公时长等
  - 写：下一次提醒时间、临时休息、策略强度、当日“宽容额度”等
- 边界（Policy 约束）：
  - 每日可暂停总额上限
  - 连续忽略超过 N 次后，申诉默认不通过或仅给最小额度
  - 站立办公可按比例替代部分站立要求（需要定义规则）
- 可控性：所有“写”操作都应可解释、可撤销，必要时要求用户点击采纳才生效。

### Phase 3：多层级干预（降低打断，提高坚持）
- 干预分级：轻提示（不抢焦点）→ 强提示（置顶但不遮挡）→ 处刑弹窗（最后手段）。
- AI 根据上下文选择等级：休息/站立办公/刚结束站立/短会议等。

### Phase 4：复盘与成长（长期价值）
- 日报/周报：识别久坐高峰、申诉高峰、最易失败时段。
- 自动建议：调整休息时段/提醒间隔/站立时长/策略阈值（需用户采纳或可回滚）。

## 需要持续对齐的产品决策（讨论清单）
- 产品风格：强制监督型 vs 协商辅助型（默认人格/策略偏向）
- 申诉成本：是否必须交换条件（替代动作或额度消耗）
- 站立办公计入规则：是否算完成站立？按多少比例折算？
- AI 自动改参数权限：全自动 vs 仅建议（需手动采纳）
- 上下文扩展：是否接入日历/专注/系统空闲等（隐私与权限边界）
- 北极星指标：最长连续久坐、站立分钟、忽略次数、或综合评分
